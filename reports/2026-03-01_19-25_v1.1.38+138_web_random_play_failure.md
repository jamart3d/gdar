# Issue Report: Web UI Random Play Initial Failure

- **App Version**: `1.1.38+138`
- **Current Time**: `2026-03-01 19:25`
- **Platform**: Web (Desktop Chrome/Edge)

## Issue Summary
When a user triggers "Play Random" (Dice icon) for the first time after loading the web application, the UI correctly selects a random show and track (updating metadata), but audio playback often fails to start. Subsequent attempts or manual track selection typically work correctly.

## Files Affected
- [gapless_audio_engine.js](file:///c:/Users/jeff/StudioProjects/gdar/web/gapless_audio_engine.js)
- [audio_provider.dart](file:///c:/Users/jeff/StudioProjects/gdar/lib/providers/audio_provider.dart) (Contextual flow)

## Potential Causes

### 1. AudioContext Resume Race Condition (High Confidence)
In `gapless_audio_engine.js`, the `api.play()` method contains logic to resume a `suspended` `AudioContext`. However, it has a logic bug:
- When `api.play()` is called while the context is suspended, it calls `_ctx.resume()`.
- Upon successful resume, it attempts to recursively call `api.play()` using the check: `if (_playing) api.play();`.
- **The Bug**: `_playing` is initialized to `false` and is only set to `true` inside `_startTrack()` (after decoding). Therefore, during the first interaction, `_playing` is still `false` when the resume promise settles, and the actual track loading/playback logic never executes.

```javascript
// gapless_audio_engine.js (Current Buggy Logic)
play: function () {
  _ensureContext();
  if (_ctx.state === 'suspended') {
    if (!_ctx._isResuming) {
      _ctx._isResuming = true;
      _ctx.resume().then(() => {
        _ctx._isResuming = false;
        if (_playing) api.play(); // _playing is false here on first run!
      });
    }
    return;
  }
  // ... actual track loading logic unreachable until resumed ...
}
```

### 2. User Gesture Expiration
While the "Dice" click provides a valid user gesture, the asynchronous nature of `setPlaylist` followed by `play` (and potential `AudioContext` resumption) can occasionally exceed the browser's strict window for starting audio, though this is less likely to be the primary cause than the logic error above.

## Proposed Solutions / Next Steps

### 1. fix `api.play` to set playback intent flag earlier
The `_playing` flag (or a new `_playIntent` flag) should be set to `true` as soon as `api.play()` is called, ensuring that the recursive call after `resume()` is triggered.

```javascript
play: function () {
  _ensureContext();
  _playing = true; // Mark intent immediately
  if (_ctx.state === 'suspended') {
    // ... resume logic ...
    // then() { if (_playing) api.play(); } will now work.
  }
}
```

### 2. Add Logging for Autoplay Failures
Update `gapless_audio_engine.js` to catch and log `NotAllowedError` in `_ctx.resume()` and track playback starts to provide better diagnostics in the field.

### 3. Verify initialize timing
Ensure `GaplessPlayer.init()` is robustly called during the first frame of the app, though the `click` handler already provides a fallback.
