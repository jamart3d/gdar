# Investigation Report: Animated RGB Animation "Stops" in Web UI

**Date:** 2026-03-01  
**Project:** GDAR Audio Player (v1.1.38+138)  
**Issue:** Reports of the animated RGB border intermittently freezing/stopping, particularly in the Web UI.

---

## 1. Executive Summary
Investigation reveals a critical lifecycle violation in `AnimatedGradientBorder.dart`. The component currently disposes of its fallback `AnimationController` inside the `build()` method. This "side-effect-on-build" pattern is highly unstable in Flutter; especially when triggered by frequent state changes from `AudioProvider` (playback progress) or `SettingsProvider`.

## 2. Technical Root Causes

### A. Lifecycle Violation: Mid-Build Disposal (Primary Suspect)
In `AnimatedGradientBorder._AnimatedGradientBorderState.build`, the following logic exists:
```dart
if (globalAnimation != null) {
  _localController?.dispose(); // Illegal disposal during build
  _localController = null;
}
```
If the "Global Clock" (`RgbClockWrapper`) lookup fluctuates—which happens frequently during navigation or background tab resumes—the component attempted to nuke its local ticker while the framework was still trying to paint the current frame. This can cause the `AnimatedBuilder` to lose its ticker source, leading to a "frozen" animation.

### B. Global Clock Lookup Flaws
The current implementation uses a `try-catch` block around `Provider.of` to find the global clock. 
- **The Issue:** `Provider.of` with `listen: false` doesn't establish a stable dependency. If the provider is temporarily unavailable (e.g., during a `Navigator` pop/push where the context is in a transition state), the widget falls back to a local controller.
- **The "Stuck" Cycle:** Rapidly switching between local/global clocks due to context shifting causes the animation to reset to `0.0` repeatedly, making it look "stopped."

### C. Browser Ticker Throttling
Chrome on Windows aggressively throttles `requestAnimationFrame` if a "Simple Theme" (Performance Mode) is not active but the GPU is under load. Although we optimized shadows out during playback, the existence of many `CustomPaint` objects in a list can still trigger this.

## 3. Recommended Fixes

1.  **Stop Disposal in Build**: Move the controller management to `didChangeDependencies` or `didUpdateWidget`. Never call `dispose()` inside `build()`.
2.  **Stabilize Clock Selection**: Use a more robust `watch` or `select` pattern for the global clock to ensure the widget has a consistent animation source throughout its lifecycle on a specific screen.
3.  **Local Controller Guard**: Ensure `_ensureLocalController()` is only called once per lifecycle if needed, rather than potentially recreating it during build-driven logic.
