# Issue Report: Web PWA Media Notification Reliability

- **App Version**: v1.3.89+299
- **Timestamp**: 2026-04-11 13:43 (PDT)
- **Status**: Investigation Complete

## Issue Summary
In the GDAR Web UI (PWA), the connection between the OS-level notification player (Media Session API) and the core application logic (Flutter/Dart) exhibits points of failure that can lead to unresponsive play/pause controls, missing skip functionality, or UI/playback state desynchronization. This is particularly prevalent during background execution on mobile browsers where OS throttling is aggressive.

## Files Affected
- `apps/gdar_web/web/audio_mediasession.js`: Centralized manager for MediaSession API.
- `apps/gdar_web/web/hybrid_audio_engine.js`: Manages active engines and action handlers.
- `apps/gdar_web/web/audio_heartbeat.js`: Survival mechanism for background execution.
- `packages/shakedown_core/lib/audio/web_interop_web.dart`: Dart-JS bridge for MediaSession.

## Architectural Investigation Findings

### 1. Missing Action Handlers (UX Weak Point)
`audio_mediasession.js` defines support for `seekbackward` and `seekforward` (lines 98-99), but `hybrid_audio_engine.js` only provides callbacks for `onPlay`, `onPause`, `onNext`, `onPrevious`, and `onSeekTo` (lines 824-830).
- **Impact**: The skip buttons (usually 10s or 30s) in the OS notification/lock-screen are non-functional or invisible because their handlers are never registered.

### 2. Engine Handoff Race Conditions
The "Hybrid" engine swaps between HTML5 (Background) and Web Audio (Foreground) to balance performance and battery.
- **Vulnerability**: Every time `_swapEngine` is called, `_syncMediaSession()` is triggered, which resets the "de-duplication" guards and re-registers all action handlers (line 813).
- **Risk**: If a user interaction (like clicking Pause) happens exactly during a `setTimeout` based handoff (e.g., `_executeForegroundRestore`), the `_handoffRunId` logic may cancel the handoff, but the MediaSession might have already been updated with the "next" engine's intent, leading to a momentary UI "jump" or a failed command if the engine being paused is not the one the callback is currently targeting.

### 3. State De-duplication Staleness
`audio_mediasession.js` uses `_lastMetadata` and `_lastPlaybackState` (lines 13-14) to avoid redundant browser writes.
- **Weak Point**: Some mobile browsers (iOS Safari) occasionally "reset" the Media Session metadata without a JavaScript-visible event if another short-lived system audio event occurs. Since GDAR thinks it's already pushed the state, it won't push it again until the metadata *actually* changes.
- **Proposed Fix**: Add a `forceSync` call triggered by common AppLifecycle transitions or "Audio Interruption" events to ensure the OS UI is always authoritative.

### 4. Background Throttling Hurdles
`audio_heartbeat.js` attempts to trick the OS into keeping the tab alive using silent audio/video.
- **Reliability Issue**: If the browser's "Auto-play" policy blocks the heartbeat (which it can if the user hasn't interacted with the page *after* it was backgrounded or if the OS is in Low Power Mode), the app will be suspended. When the user later clicks "Play" on the notification, the JS context might be too "frozen" to respond in time, causing the OS to report a "Playback Error" and clear the notification.

## Potential Causes
1.  **Incomplete Handler Registration**: Failure to map all supported Media Session actions to the Hybrid API.
2.  **Engine Boundary Latency**: The 250ms/100ms polling in `_executeForegroundRestore` creates a window where the App and Notification UI are not 100% in sync.
3.  **OS Throttling**: Aggressive PWA cleanup on iOS/Android.

## Proposed Solutions / Next Steps

### Short-Term (Reliability Fixes)
1.  **Register Skip Handlers**: Update `_setupMediaSession` in `hybrid_audio_engine.js` to handle `onSeekBackward` and `onSeekForward`.
2.  **Heartbeat Resilience**: Add a "Heartbeat Failed" telemetry event in `audio_heartbeat.js` so we can track how often the survival trick is being blocked by the browser.

### Long-Term (Architectural Improvements)
1.  **Centralised Command Gateway**: Move Media Session action handling to a more resilient "Worker" or a dedicated singleton that is less affected by engine-swaps.
2.  **OS Feedback Loop**: Implement a periodic "Pulse" where the app re-broadcasts its state to MediaSession every 10-15 seconds regardless of "de-duplication" logic, specifically when hidden.
3.  **Handoff Atomicity**: Refactor engine swaps to be more atomic, ensuring action handlers are never in an "indeterminate" state during a handoff.
