# Issue Report: Fruit Car Mode Playback Stall (Web/PWA)

- **Report Date**: 2026-04-05 14:46
- **App Version**: v1.3.61+271
- **Platform**: Web (Fruit UI)
- **Status**: Investigated

## Issue Summary
In the "Fruit" style car mode playback screen on the Web/PWA platform, the current duration and progress bar sometimes stop updating. The user reports that starting a new playlist resolves the issue, suggesting a stall in the underlying audio engine's communication or heartbeat mechanism.

## Files Affected
- `packages/shakedown_core/lib/ui/screens/playback_screen_fruit_car_mode.dart` (UI layer)
- `packages/shakedown_core/lib/services/gapless_player/gapless_player_web_engine.dart` (Bridge layer)
- `packages/shakedown_core/lib/providers/audio_provider_playback.dart` (Orchestration layer)
- `hybrid_init.js` (Web Audio Engine asset - indirectly)

## Potential Causes

### 1. JS-to-Dart Heartbeat Failure
The `GaplessPlayerWeb` depends on a `_onJsStateChange` callback triggered from the JavaScript side (`window._gdarAudio`). If the JS engine (which typically uses `requestAnimationFrame` or `setInterval` for its ticker) stops calling this callback, the Dart broadcast streams (`positionStream`, `durationStream`) will stop emitting events, freezing the UI.

### 2. Browser/PWA Throttling
Modern browsers aggressively throttle background tabs. If the Fruit UI/PWA determines it is not the primary focus, it might deprioritize the JS execution loop. Since the PROGRESS depends on active "ticks" from JS, the UI remains frozen at the last known position even if audio continues playing (which often happens as audio is handled by a separate process/thread in the browser).

### 3. State Inconsistency ("requires new playlist")
The fix ("new playlist") involves calling `setPlaylist` on the JS engine. This likely force-restarts the engine's internal loops. A stall might occur if the engine enters a state like `suspended_by_os` or an unhandled buffer exhaustion that it fails to transition out of correctly once data is available again.

## Proposed Solutions

### A. Implement a Dart-side Watchdog
Add a monitoring timer in `GaplessPlayerWebEngine` that checks if `playing` is true but no ticks have been received for a threshold (e.g., 2 seconds). If a stall is detected, the Dart side should "ping" the JS engine for an update or attempt a silent resume.

### B. Resilience to Visibility Changes
Force a state re-synchronization when the tab becomes `visible` using the existing visibility listener in `_GaplessPlayerWebEngine`.

### C. Watchdog notification in Car Mode
Since Car Mode is a specific "active" UI, the `AudioProvider` could increase its polling frequency or use a specific "heartbeat" stream to ensure the UI is kept alive.
