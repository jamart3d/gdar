## Verified Findings from Original Report

### 1. Missing Skip Handlers — CONFIRMED

`audio_mediasession.js:98-99` correctly lists `seekbackward` and `seekforward` in
`setActionHandlers`. However, `hybrid_audio_engine.js:822-831` (`_setupMediaSession`)
only passes five callbacks:

```js
onPlay, onPause, onNext, onPrevious, onSeekTo
```

`onSeekBackward` and `onSeekForward` are **never passed**, so inside
`setActionHandlers` the loop calls:

```js
navigator.mediaSession.setActionHandler('seekbackward', null);
navigator.mediaSession.setActionHandler('seekforward', null);
```

Setting a handler to `null` explicitly *removes* it from the OS notification UI.
The skip-10s / skip-30s buttons on the lock screen are effectively hidden on every
platform that respects this spec (Android Chrome, iOS Safari 17+).

**Fix**: Add `onSeekBackward` and `onSeekForward` callbacks in `_setupMediaSession`:

```js
onSeekBackward: (e) => api.seek(Math.max(0, _currentPosition - (e.seekOffset || 10))),
onSeekForward:  (e) => api.seek(_currentPosition + (e.seekOffset || 10)),
```

`_currentPosition` must be tracked in the hybrid engine (or fetched from
`_activeEngine.getState().position`).

### 2. Engine Handoff Race — CONFIRMED

`_swapEngine` (line 813) calls `_syncMediaSession()` synchronously. `_syncMediaSession`
calls `forceSync()` which zeroes `_lastMetadata` and `_lastPlaybackState`, then
re-registers action handlers. If a Dart-originated `updatePlaybackState` arrives via
`WebInterop.syncMediaSession` (Dart `web_audio_engine.dart:79`) **concurrently** with
a handoff, two writers briefly contend on the browser's MediaSession state. The `forceSync`
de-dup reset means the Dart call will always write through, potentially overwriting
the handoff's intent.

The `_handoffRunId` guard (engine swap cancellation) only protects the engine swap
logic itself — it does not prevent stale MediaSession writes from a racing Dart call.

---

### 3. De-duplication Staleness — CONFIRMED

`forceSync()` exists and is called on every `_swapEngine`. There is currently no
periodic "pulse" while the tab is hidden. The proposed fix (re-broadcast every 10–15s
while `document.visibilityState === 'hidden'`) is sound and not yet implemented.

---

### 4. Background Throttling Hurdles — CONFIRMED

`audio_heartbeat.js:59-62` swallows failures silently:

```js
_heartbeatAudio.play().catch(err => {
    _log.log('[gdar heartbeat] Audio prime failed (handled):', err.message);
});
```

There is no counter, no telemetry emission, and no way for Dart or the HUD to know
the heartbeat failed. The report's recommendation to add a failure telemetry event
is valid and not yet implemented.

## New Findings (Code-Verified)

### 5. `WebInterop.syncMediaSession` Only Bridges Playback State

`web_interop_web.dart:5-17` binds exactly one JS function:

```dart
@JS('window._gdarMediaSession.updatePlaybackState')
external void _jsUpdatePlaybackState(bool playing);
```

The Dart layer has **no binding** for `updateMetadata` or `updatePositionState`.
This means:
- If a future Dart-driven path (non-hybrid engine, passive engine, or new engine)
  needs to set track metadata, it has no available bridge — only the JS engines can
  write metadata.
- The stub (`web_interop_stub.dart:4`) is a no-op for all platforms; non-web paths
  are safe.

**Risk**: If the Dart audio provider ever needs to push metadata (e.g., chapter titles,
stream metadata not known at queue-load time), there is no path to do so without
adding a new `@JS` binding.

---

### 6. Heartbeat `isActive()` Masks Audio Failure

`audio_heartbeat.js:99-101`:

```js
isActive: function () {
    return !!(this.isAudioActive() || this.isVideoActive());
}
```

The OR means: if audio heartbeat fails (autoplay blocked) but the video heartbeat
succeeds, `isActive()` returns `true`. The HUD state key `heartbeatActive` will show
green even though the primary survival mechanism is down. This can mask chronic audio
autoplay failures on iOS Low Power Mode, where video may also be blocked minutes later.

**Suggestion**: Expose `isAudioActive()` and `isVideoActive()` separately in the state
object (or add a `heartbeatMode: 'audio'|'video'|'both'|'none'` key) so the HUD can
reflect the actual fallback tier.

---

### 7. Heartbeat Audio Volume Is Not Zero

`audio_heartbeat.js:31`:

```js
_heartbeatAudio.volume = 0.01;
```

`0.01` is not silent — it is approximately −40 dBFS. On high-gain headphone setups
or quiet environments this may be audible as a faint hiss when the silent WAV loops.
Volume should be `0` (true silence) since the OS keeps the tab alive based on the
_element playing_, not on audible output.

**Fix**: Change to `_heartbeatAudio.volume = 0;`

Note: Some older browsers historically required `volume > 0` to count as "playing".
This should be verified against current Chrome/Safari behaviour before changing.

---

### 8. `onStop` Handler Is Never Registered

`_setupMediaSession` provides no `onStop` callback. `audio_mediasession.js:97`
includes `['stop', callbacks.onStop]` in the action list, so like the seek handlers,
this is set to `null` → removed from the OS UI. On platforms that show a "stop" button
(some Android lock screen widgets), it will be absent. This is likely intentional for
a streaming music player, but should be documented.

---

### 9. `_syncMediaSession` Doesn't Re-push Metadata

`_syncMediaSession()` (line 834-842) resets dedup, re-registers handlers, pushes
playback state and position state — but **does not call `updateMetadata`**. If an
engine handoff occurs while a track is playing, the OS notification title/artist
remain from the previous push. After a handoff, `_lastMetadata` is zeroed (by
`forceSync`) so the next `updateMetadata` call will write through — but nothing
*triggers* that call. Metadata is only pushed reactively (when the track changes).
If a handoff occurs mid-track, no metadata update is sent.

**Risk**: Low in practice (handoffs are intra-track), but the metadata is guaranteed
stale until the next track change after a handoff.

**Fix**: After `_syncMediaSession`, call `updateMetadata` with the current track's
cached metadata from the hybrid engine's internal `_currentMetadata` (if tracked).

## Recommended Fix Order

1. **Finding 1** — register seek handlers. Zero risk, clear UX win.
2. **Finding 7** — set volume to 0. One-liner, verify in Chrome/Safari first.
3. **Finding 9** — re-push metadata in `_syncMediaSession`. Low complexity.
4. **Finding 4** — add heartbeat failure telemetry. Improves observability.
5. **Finding 6** — expose tiered heartbeat state in HUD.
6. **Finding 3** — add hidden-tab pulse (10–15s interval, clear on foreground).
7. **Findings 2/5** — defer until a Dart-driven metadata path is needed.
