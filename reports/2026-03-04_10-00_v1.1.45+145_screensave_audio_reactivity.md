# Issue Report: Screensaver & Audio Reactivity Analysis

**Date:** 2026-03-04  
**Time:** 10:00 AM  
**App Version:** 1.1.45+145

---

## 1. Issue Summary
The "Steal Your Face" screensaver's audio reactivity is currently non-functional across all target platforms. While the visual engine (Flame + GLSL Shaders) is built to receive and respond to audio energy data, the data pipeline is disconnected or blocked at the source.

---

## 2. Investigation Findings

### Android TV (Google TV)
The system uses the native Android `Visualizer` API via the `VisualizerAudioReactor` class. However, it fails to function because:
- **Missing Permission**: The Android Visualizer API requires the `RECORD_AUDIO` permission to monitor the system's output mix.
- **Dependency Missing**: `permission_handler` is not present in `pubspec.yaml`, and the code never requests permission from the user.
- **Stale Settings**: If `pulseIntensity` is `0.0` (default in some modes), the visual reactivity is invisible even if the data were flowing.

### Web (PWA)
The `AudioReactorFactory` currently returns `null` for Web platforms. 
- **Denied by Factory**: `lib/visualizer/audio_reactor_factory.dart` explicitly checks `if (!isTv || kIsWeb) return null;`.
- **Missing Implementation**: There is no `WebAudioReactor` class to interface with the Browser's `AudioContext.createAnalyser()` node.

### Phone (Mobile)
Like Web, reactivity is explicitly disabled in the factory for phone devices to prioritize performance.

---

## 3. Files Affected
- **Infrastructure**: `pubspec.yaml` (missing permissions)
- **Integration**: `lib/visualizer/audio_reactor_factory.dart` (factory logic)
- **Native Implementation**: `lib/visualizer/visualizer_audio_reactor.dart`
- **UI Logic**: `lib/ui/screens/screensaver_screen.dart` (needs to request permission)
- **Configuration**: `lib/config/default_settings.dart` (pulse default values)

---

## 4. Potential Causes
1. **Security Policy**: Android's strict hardware permissioning.
2. **Platform Parity Gap**: The focus was primarily on Android TV's native visualizer, leaving Web/Phone as "pass-through" null implementations.
3. **Implicit Initialization**: The visualizer expects an `audioSessionId` which may not always be reliably passed from the `GaplessPlayer` service.

---

## 5. Proposed Solutions & Next Steps

### Step 1: Fix Android TV (Immediate)
1. Add `permission_handler: ^11.3.1` to `pubspec.yaml`.
2. Update `lib/ui/screens/screensaver_screen.dart` to request the `audio` permission before calling `AudioReactorFactory.create()`.
3. Modify `SettingsProvider.toggleOilEnableAudioReactivity()` to ensure `oilPulseIntensity` is at least `1.0` (otherwise the reactivity is invisible).

### Step 2: UI Feedback
1. Ensure the "Audio Reactivity" settings are clearly labeled as TV-only or hidden on other platforms to avoid confusion.
