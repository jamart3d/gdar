# Issue Report: Web Audio Engines Deep Audit

**Date:** 2026-03-11
**App Version:** v1.1.66+166
**Status:** Deep Investigation Complete

---

## Issue Summary
This audit evaluated the performance, reliability, and architectural integrity of GDAR's triple-engine web audio stack:
1.  **Orchestrator**: `hybrid_audio_engine.js` (The brain)
2.  **Foreground**: `gapless_audio_engine.js` (Pure Web Audio)
3.  **Background**: `hybrid_html5_engine.js` (Relisten Gapless Port)
4.  **Survival**: `audio_heartbeat.js` (Background maintenance)

While the stack handles browser-specific restrictions well (using the Hybrid Escape Hatch), significant risks were identified in memory management, background throttling, and decoder latency.

## Files Affected
- `web/hybrid_audio_engine.js`
- `web/hybrid_html5_engine.js` (Relisten Port)
- `web/gapless_audio_engine.js`

## Detailed Findings

### 1. Relisten Engine (Hybrid HTML5) - Memory Management Risk
- **Issue**: `PRELOAD_NUM_TRACKS` is hardcoded to `5`. 
- **Cause**: The `loadBuffer()` method recursively preloads up to 5 tracks as full Web Audio buffers (PCM). 
- **Impact**: On high-resolution tracks (FLAC/MP3) typical of Grateful Dead shows (10-20 mins each), this can easily consume 500MB+ of RAM. Mobile browsers (Safari/iOS) will frequently kill the tab for OOM (Out Of Memory) violations.
- **Risk Level**: **CRITICAL** for mobile stability.

### 2. Background Throttling (requestAnimationFrame Dependency)
- **Issue**: In `hybrid_html5_engine.js`, the `onProgress` loop is powered by `window.requestAnimationFrame`.
- **Cause**: This loop triggers the "pre-fetch" of the next track when a track has 25 seconds remaining.
- **Impact**: When the browser tab is backgrounded, `requestAnimationFrame` is severely throttled or paused. This may result in the next track never pre-fetching, leading to a silence/gap when the current track finishes.
- **Mitigation**: The Orchestrator uses a `Worker`, but the internal Relisten engine logic is still tied to the UI loop.
- **Risk Level**: **HIGH** for gapless reliability in background.

### 3. Decoder & Network Latency (Gapless/WebAudio)
- **Issue**: No "streaming" support in custom WA engines.
- **Cause**: Both `gapless_audio_engine.js` and the Relisten port fetch the *entire* file before decoding and starting playback.
- **Impact**: For 20-minute tracks on slow networks, the UI remains in a "buffering" state for an extended period even though standard HTML5 could have started playing within seconds via range-requests.
- **Risk Level**: **MEDIUM** for user experience.

### 4. Logic Brittle-ness in Hybrid Orchestrator
- **Issue**: Hardcoded engine technology strings (e.g., `hybrid_(WA)`, `hybrid_(H5)`) and inconsistent `contextState` reporting.
- **Risk Level**: **LOW** (Maintenance debt).

---

## Proposed Solutions / Next Steps

### 1. Reduce Memory Footprint
- **Action**: Lower `PRELOAD_NUM_TRACKS` from `5` to `1` or `2` in `hybrid_html5_engine.js`.
- **Action**: Implement a "Release Cache" mechanism to clear buffers of tracks that have already played + 2.

### 2. Decouple Timing from UI Loop
- **Action**: Migrate the `onProgress` trigger in `hybrid_html5_engine.js` to listen to the `gdar-worker-tick` CustomEvent instead of `requestAnimationFrame`. This ensures pre-fetches trigger even when backgrounded.

### 3. Progressive Loading (Long Term)
- **Action**: Investigate using MediaSource API or chunks for the Web Audio engines to allow "early start" while still maintaining the gapless sample-accuracy for the transition.

### 4. Dynamic Stall Recovery
- **Action**: Broaden the 5-second stall recovery to monitor generic `currentTime` progress failures, not just the explicit `buffering` state, to catch "silent hangs."
