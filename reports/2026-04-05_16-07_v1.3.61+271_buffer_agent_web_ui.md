# Issue Report: Buffer Agent Behavior in Web UI

## Issue Summary
The `BufferAgent` is an intelligent playback recovery service within the `shakedown_core` package designed to monitor and recover from audio buffering stalls. This report investigates its operational status and effectiveness specifically within the Web (PWA/Fruit) target of the GDAR application.

## Investigation Results
The investigation confirms that the **Buffer Agent is fully active and functional in the Web UI**.

### 1. Active Status
`BufferAgent` is enabled by default in `DefaultSettings.enableBufferAgent` and is correctly instantiated by the `AudioProvider` on web. It successfully:
- Subscribes to the `GaplessPlayer`'s `playerStateStream` and `playbackEventStream`.
- Monitors network connectivity via `connectivity_plus` (mapping to `navigator.onLine`).
- Observes lifecycle changes (visible vs hidden tabs).

### 2. Operational Logic on Web
- **Stall Detection**: If the player processing state enters `buffering` and stays there for **20 seconds**, the agent triggers recovery.
- **Recovery Method**: It attempts a `seek` to the current position followed by a `play()` call.
- **Background Logic**: Unlike the web-specific `WebTickStallPolicy` (which only resyncs if the tab is visible), `BufferAgent` continues to monitor in the background, though browser timer throttling may delay its execution.

## Potential Issues & Constraints
While functional, several "web-isms" may impact its reliability:

| Factor | Behavioral Delta on Web |
| :--- | :--- |
| **Auto-Play Policies** | Browsers may block the recovery `play()` call if the tab has been in the background or hasn't had recent user interaction ("Sticky Activation" loss). |
| **Timer Throttling** | When the PWA is hidden, the 20s threshold may effectively become 60s+ because `Timer.periodic` is deprioritized by JS engines. |
| **Connectivity Accuracy** | `connectivity_plus` on web only reports the browser's global online status; it cannot detect if the specific MP3 stream is being throttled or blocked by a CDN. |
| **Engine Redundancy** | The `GaplessPlayer` JS engine has its own `stale_tick` watchdog. `BufferAgent` and the watchdog occasionally perform similar "resync" actions, which is safe but redundant. |

## Proposed Solutions / Next Steps

### 1. Auto-Play Guard (High Priority)
Update the `BufferAgent._performRecovery` or the calling `AudioProvider` to detect if the browser blocked the playback resumption.
> [!TIP]
> Catch the `play()` Promise/Future error on web. If it's an `AllowedException` or similar, show a "Tap to Resume" UI prompt instead of failing silently.

### 2. Web-Specific Threshold Tuning
Consider shortening the buffering threshold from 20s to **10s** for web users on the `auto` engine. Web fetch failures are often immediate and permanent rather than transient (unlike mobile LTE where signal might fluctuate).

### 3. Visibility Integration
Ensure `BufferAgent` uses its `isAppVisible` check to choose between:
- **Foreground**: Immediate recovery with a UI notification ("Network issue detected...").
- **Background**: Silent recovery attempt (existing behavior).
