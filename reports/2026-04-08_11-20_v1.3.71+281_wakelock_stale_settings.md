# Issue Report: PWA Wake Lock Persistence Bug

**Date:** 2026-04-08  
**Time:** 11:25 AM  
**App Version:** 1.3.71+281 (Web PWA)  

## Issue Summary
In the Web UI (Fruit theme), toggling the "Keep Screen On" setting to **OFF** while audio is currently playing fails to release the device wake lock. The screen remains active even though the UI indicates the setting is disabled.

## Files Affected
- [audio_provider_lifecycle.dart](file:///home/jam/StudioProjects/gdar/packages/shakedown_core/lib/providers/audio_provider_lifecycle.dart)
- [wakelock_service.dart](file:///home/jam/StudioProjects/gdar/packages/shakedown_core/lib/services/wakelock_service.dart) (Indirectly)

## Potential Causes
The primary cause is an **ordering bug** in the `AudioProviderLifecycle.update()` method. 

In `AudioProviderLifecycle.update()`, the logic to detect a change in the `preventSleep` setting and trigger a wake lock update runs **before** the local `_settingsProvider` reference is updated.

```dart
// packages/shakedown_core/lib/providers/audio_provider_lifecycle.dart

132:     if (_lastPreventSleep == null ||
133:         settingsProvider.preventSleep != _lastPreventSleep) {
134:       _lastPreventSleep = settingsProvider.preventSleep;
135:       _updateWakeLockState(); // <--- This call uses the STALE _settingsProvider!
136:     }
137: 
138:     _settingsProvider = settingsProvider;
```

Inside `_updateWakeLockState()`:
```dart
57:   Future<void> _updateWakeLockState() async {
58:     final shouldPreventScreensaver = _settingsProvider?.preventSleep ?? true;
    ...
```

When the user toggles the setting:
1. `update()` is called with the NEW `settingsProvider`.
2. `settingsProvider.preventSleep` is `false`.
3. `_lastPreventSleep` (old value) was `true`.
4. `_updateWakeLockState()` is triggered.
5. `_updateWakeLockState()` reads `_settingsProvider?.preventSleep`. Since `_settingsProvider` has not been updated yet (happens on line 138), it reads the **old value** (`true`).
6. Because `isPlaying` is likely `true` during this toggle, the logic concludes that the wake lock should still be active and calls `enable()` again instead of `disable()`.
7. `_settingsProvider` is finally updated, but the wake lock remains active.

## Proposed Solutions / Next Steps

### 1. Fix Order of Operations (Immediate)
Update `_settingsProvider` reference **before** calling `_updateWakeLockState()` or any other methods that dependency on current settings.

```diff
-    if (_lastPreventSleep == null ||
-        settingsProvider.preventSleep != _lastPreventSleep) {
-      _lastPreventSleep = settingsProvider.preventSleep;
-      _updateWakeLockState();
-    }
-
     _settingsProvider = settingsProvider;
+
+    if (_lastPreventSleep == null ||
+        settingsProvider.preventSleep != _lastPreventSleep) {
+      _lastPreventSleep = settingsProvider.preventSleep;
+      _updateWakeLockState();
+    }
```

### 2. Pass Explicit State to Update Method (Robust)
Modify `_updateWakeLockState()` to accept the `preventSleep` value as an argument to avoid confusion between current and cached provider state.

### 3. Verification
Run the existing regression tests in `packages/shakedown_core/test/services/web_gapless_adapter_test.dart` and add a new test case specifically for toggling the setting *during* active playback.
