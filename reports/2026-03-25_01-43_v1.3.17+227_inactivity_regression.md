# Issue Report: InactivityService Regression (TV Screensaver)

**Date**: 2026-03-25  
**Time**: 01:43  
**App Version**: v1.3.17+227  
**Status**: Investigated / Solution Proposed

---

## 1. Issue Summary
The `TV Startup Regressions` tests for `gdar_tv` are failing because the screensaver inactivity timeout never fires. Specifically, the test `TV app launches screensaver after inactivity when enabled` times out during the `tester.pump(duration)` call, despite the timer ticking as expected. This regression was introduced by commit `b3ef700`.

## 2. Files Affected
- `packages/shakedown_core/lib/services/inactivity_service.dart`
- `apps/gdar_tv/test/tv_startup_regression_test.dart` (Test environment)

## 3. Root Cause Investigation
Commit `b3ef700` converted the `InactivityService` from a **one-shot timer** to a **periodic polling timer** (`Timer.periodic`) running every 1 second. 

- **Code Logic**: Inside the `_tick` loop, the service uses `DateTime.now().difference(_lastActivityAt)` to calculate elapsed time.
- **The Failure**: In the Flutter test environment, `tester.pump(const Duration(minutes: 1, seconds: 5))` advances the virtual clock for the `FakeAsync` scheduler. The `Timer.periodic` fires 65 times during this pump.
- **The "Black Box" Bug**: However, in standard `flutter_test` (without explicit virtualization via `package:clock`), `DateTime.now()` returns the **real-world time**. 
- **The Result**: All 65 ticks of the polling timer occur in a tight real-world loop (taking only milliseconds). During these milliseconds, `DateTime.now()` does not advance enough to reach the 1-minute threshold. The `elapsed` value remains `0s` for every tick, thus `remaining` is always `59s`, and the timeout never triggers.

**Logs during test**:
```text
InactivityService: start (1m timeout)
InactivityService: tick elapsed=0s, remaining=59s, timeoutFired=false
InactivityService: tick elapsed=0s, remaining=59s, timeoutFired=false
... [x65]
```

## 4. Potential Solutions

### A. Virtualize Time (Recommended)
Add `package:clock` to `shakedown_core` and replace all calls to `DateTime.now()` in `InactivityService` with `clock.now()`. This ensures that in tests, the service picks up the `FakeAsync` advancement provided by `tester.pump()`.

### B. Dependency Injection
Modify the `InactivityService` constructor to accept a `DateTime Function() now` callback, defaulting to `DateTime.now`.
```dart
class InactivityService {
  final DateTime Function() _now;
  InactivityService({ ..., DateTime Function()? now }) : _now = now ?? () => DateTime.now();
}
```

### C. Fallback for Tests
If `kDebugMode` is active, or via a setter, allow the `InactivityService` to use a manual manual clock. However, Solution A is the standard Flutter pattern for this issue.

---

## 5. Next Steps
1. Add `package:clock` to `packages/shakedown_core/pubspec.yaml`.
2. Refactor `InactivityService` to use `clock.now()`.
3. Rerun `melos run test` to verify the fix across all monorepo targets.
