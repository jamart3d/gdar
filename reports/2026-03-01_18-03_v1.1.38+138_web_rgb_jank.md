# Issue Report: Web UI RGB Animation Jank During Playback

**Date:** 2026-03-01  
**App Version:** 1.1.38+138  
**Environment:** Web (Chrome on Windows), `flutter run -d chrome`

---

## 1. Issue Summary
In the Flutter Web UI, animated RGB elements (such as `AnimatedGradientBorder`) and Liquid Glass effects become janky and drop frames specifically when a track is playing. When audio is paused or stopped, the animations return to a smooth 60fps state.

## 2. Files Affected
- `lib/ui/widgets/animated_gradient_border.dart`
- `lib/ui/widgets/theme/liquid_glass_wrapper.dart`
- `lib/ui/widgets/rgb_clock_wrapper.dart`
- `lib/ui/widgets/playback/playback_panel.dart`

## 3. Potential Causes
Based on investigation, the root cause is **graphical resource contention** on the main thread/GPU when audio playback is active.

### Key Factors:
- **Global Clock Rebuilds**: `RgbClockWrapper` provides a global `AnimationController` that triggers 60fps rebuilds for all `AnimatedGradientBorder` widgets using `AnimatedBuilder`.
- **Expensive Web Blurs**: 
    - `AnimatedGradientBorder` uses `MaskFilter.blur(BlurStyle.normal, 12)` in its `CustomPaint`.
    - `LiquidGlassWrapper` (used in the `PlaybackPanel`) uses `BackdropFilter` with a high sigma (32.0). 
    - On Flutter Web (Canvaskit/HTML), these blur operations are computationally heavy.
- **Audio Engine Overhead**: While the Web build correctly disables the real-time `AudioReactor` (FFT analysis), the act of playing audio (decoding, managing streams, and OS-level communication) consumes significant CPU/Main Thread slices.
- **Threshold Effect**: The system has enough headroom to render these blurs at 60fps when idle, but the added load of the audio engine pushes it over the frame budget (~16.6ms), resulting in dropped frames (jank).

## 4. Proposed Solutions / Next Steps
1.  **Lower Refreshrate during Playback**: 
    - Consider reducing the animation frequency or using a stepped animation (e.g., 30fps) instead of a raw 60fps clock when audio is active.
2.  **Optimize Blurs for Web**:
    - Reduce the `glowOpacity` or the `blur` sigma on Web builds specifically.
    - Leverage `Performance Mode` more aggressively on Web to disable blurs when playback is active if the device is under load.
3.  **Conditional Throttling**:
    - Use `Stream.throttle` or similar logic if any UI elements (like progress bars) are updating too frequently during playback. Currently, `AnimatedGradientBorder` is the most likely culprit due to its per-frame ticker.
4.  **Static Fallbacks**:
    - On Web, replace the expensive `CustomPaint` blur with a simple pre-rendered CSS shadow or a static gradient if the system detects slow frame times.
