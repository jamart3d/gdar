## 1. Issue Summary
The user reports that the screensaver's "Audio Reactivity" feature may not be working even when enabled in settings. Investigation reveals that the feature is technically functional but is restricted by strict hardware gates and sensitive default parameters that make it appear inoperable on most mobile devices.

## 2. Evidence & Observations

### A. Strict Hardware Gate (TV-Only)
In `lib/visualizer/audio_reactor_factory.dart`, the creation of a functional `AudioReactor` is strictly limited to devices where `isTv` is true:
```dart
// lib/visualizer/audio_reactor_factory.dart:17
if (!isTv || kIsWeb) {
  return null;
}
```
**Impact:** On Android Phones and Tablets, the "Enable Audio Reactivity" toggle in settings will save correctly, but the screensaver will never initialize an `AudioReactor`, resulting in zero visual response to music.

### B. Master Intensity Nullifies Reactivity
The shader `shaders/steal.frag` uses `uPulseIntensity` as a master multiplier for all audio-driven effects (bass nudge, scale pulse, treble flash).
```dart
// lib/config/default_settings.dart:80
static const double oilPulseIntensity = 0.0;
```
**Impact:** Even on a TV device, if a user enables "Audio Reactivity" but does not manually increase the "Pulse Intensity" (which defaults to 0.0), the shader multipliers will stay at zero, effectively silencing the visualization.

### C. Permission Handling Gap
The Android Visualizer API requires the `RECORD_AUDIO` permission. While it is declared in the `AndroidManifest.xml`, there is no runtime permission request logic in `ScreensaverScreen` or `VisualizerAudioReactor`.
**Impact:** On Android 6.0+ devices, the visualizer may fail to initialize silently if the permission was not manually granted in system settings.

### D. Web/PWA Incompatibility
The `ScreensaverScreen` explicitly returns early if `kIsWeb` is true:
```dart
// lib/ui/screens/screensaver_screen.dart:99
if (kIsWeb || !settings.oilEnableAudioReactivity) return;
```
**Impact:** Web and PWA versions will never have audio reactivity, regardless of settings.

## 3. Files Affected
- `lib/visualizer/audio_reactor_factory.dart`: Restricts reactor creation to TV.
- `lib/config/default_settings.dart`: Sets `oilPulseIntensity` default to `0.0`.
- `lib/ui/screens/screensaver_screen.dart`: Early-exit logic for Web/Reactivity toggle.
- `lib/visualizer/visualizer_audio_reactor.dart`: Lacks runtime permission handling.

---

## 4. Potential Causes
1.  **Intentional Restriction:** The visualizer might have been gated to TV to prevent battery drain or UI lag on mobile devices.
2.  **Missing Defaults:** `oilPulseIntensity` was likely initialized to `0.0` to ensure a "calm" experience by default, but this conflicts with the expectation that turning reactivity "On" should do *something*.
3.  **Permission Oversight:** The app assumes permissions are handled at the OS/Install level, which is not true for `RECORD_AUDIO` since API 23.

---

## 5. Proposed Solutions / Next Steps

### Phase 1: Immediate UX Fixes (Low Risk)
- **Relax Hardware Gate**: Update `AudioReactorFactory.create` to allow Android Phones/Tablets if they support the Visualizer API.
- **Update Defaults**: Set `oilPulseIntensity` default to `1.0` (Subtle) in `DefaultSettings.dart`.
- **UI Logic Update**: In `SettingsProvider`, when `toggleOilEnableAudioReactivity()` is called to enable, automatically set `oilPulseIntensity` to `1.0` if it is currently `0.0`.

### Phase 2: Robustness (Medium Risk)
- **Permission Request**: Add a `Permission.microphone.request()` (using `permission_handler`) in `ScreensaverScreen._initAudioReactor` before attempting to initialize the reactor.
- **Web Fallback**: Consider implementing a basic `PositionAudioReactor` for Web that uses the current track position/time to simulate "fake" periodic reactivity for visual interest on PWA.
